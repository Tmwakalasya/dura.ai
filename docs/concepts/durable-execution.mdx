## The dangerous window

Every process that calls external APIs has a window where it can lose state:

```
charge_card()      ← side effect committed to Stripe ($412 charged)
# 💥 OOM kill, deploy restart, pod evicted — process is gone
book_flight()      ← never runs
```

On restart, the agent has no memory of the charge. It charges again. The
customer is billed twice. The flight is still not booked.

This isn't a bug in your code. It's a fundamental property of any stateful
process that can die between steps. The dangerous work already happened — the
danger is that retrying repeats it.

The fix is not catching exceptions. A `kill -9` doesn't raise. A docker restart
doesn't raise. An OOM kill doesn't raise. You need the result committed to
durable storage *before* the process continues — so that a restart can read it
back and skip the step.

That's what dura does.

---

## The mechanism: write-ahead log

dura records the result of each step to a **write-ahead log (WAL)** *before*
the workflow is allowed to continue. The log is a local SQLite file — zero
external dependencies.

The contract for every `ctx.step("name", fn)`:

1. Look up `(run_id, "name")` in the log.
2. **Found** → return the committed result. `fn` is never called.
3. **Not found** → call `fn`, commit the result to the log, return it.

Because the commit happens before the next line of your function executes,
there is no window where a step ran but wasn't recorded.

A crash either:
- Happens **before the commit** → the step replays on next restart. Correct: nothing was recorded.
- Happens **after the commit** → the step is skipped on next restart. Correct: result is in the log.

**Either way: exactly-once.**

## Replay in detail

```python
@durable
def book_trip(ctx, dest, amount):
    charge = ctx.step("charge_card", lambda: charge_card(amount))
    flight  = ctx.step("book_flight", lambda: book_flight(dest))
    ctx.step("update_crm", lambda: update_crm(flight))
```

### First run — all three steps execute

```
WAL after run:
  charge_card  →  {"id": "ch_1", "amount": 412}   status: done
  book_flight  →  {"id": "fl_1", "dest": "Tokyo"}  status: done
  update_crm   →  {"ok": true}                     status: done
```

### Process crashes immediately after `charge_card` commits

```
WAL at time of crash:
  charge_card  →  {"id": "ch_1", "amount": 412}   status: done
  (book_flight and update_crm never started)
```

### Restart with the same arguments

dura re-runs the function body from the top. Before each step it checks the log:

| Step | In log? | What happens |
|------|---------|--------------|
| `charge_card` | ✓ yes | Returns cached result — `fn` **not called** |
| `book_flight` | ✗ no | Executes `fn`, commits result, returns it |
| `update_crm` | ✗ no | Executes `fn`, commits result, returns it |

The charge fires exactly once. The customer is never double-billed.

## The run ID

dura needs to match a restart to the right WAL entries. It derives a
deterministic **run ID** from the function name and its inputs via SHA-256:

```python
run_id = sha256(f"{fn.__name__}:{json.dumps(args)}:{json.dumps(kwargs)}")[:12]
```

Calling with the same arguments automatically resumes the same run. No explicit
ID, no infrastructure changes.

```python
book_trip("Tokyo", 412)   # run_id: "a1b2c3d4e5f6"
# crash …
book_trip("Tokyo", 412)   # same inputs → same run_id → resumes from WAL
book_trip("Nairobi", 412) # different inputs → new run_id → fresh run
```

When the same logical run might be triggered with slightly different arguments
(a retry wrapper that appends a timestamp, for example), pass a stable ID
explicitly:

```python
book_trip("Tokyo", 412, run_id="order_42")
```

## What's in the log

After any run, inspect the SQLite file directly:

```bash
sqlite3 dura.db "SELECT run_id, step_name, status, result_json FROM steps;"
```

```
a1b2c3d4e5f6  charge_card  done  {"id": "ch_1", "amount": 412}
a1b2c3d4e5f6  book_flight  done  {"id": "fl_1"}
a1b2c3d4e5f6  update_crm   done  {"ok": true}
```

Each row is one committed step. The `result_json` column is what gets returned
on replay. You can read, back up, or move this file like any other SQLite
database.

## Storage

By default dura writes to `dura.db` in the current directory. Override it:

```python
# per-call
book_trip("Tokyo", 412, log="orders/order_42.db")

# per-decorator (applies to every call)
@durable(log="/var/dura/workflows.db")
def book_trip(ctx, dest, amount): ...
```

For multiple workers sharing state, point them all at the same file on shared
storage. dura uses SQLite's WAL journal mode and a `BEGIN IMMEDIATE` write lock
on step claims, so concurrent workers are safe — only one executes any given
step, the others read the committed result.

<Note>
SQLite ships with Python's standard library — dura has **zero dependencies**.
For production multi-region deployments, dura Cloud (coming soon) replaces
the local SQLite file with a replicated managed log.
</Note>

## The SIGKILL proof

The test suite includes a real `kill -9` test — not a caught exception, an
actual uncatchable process kill mid-workflow — followed by a fresh process
restart:

```bash
python -m pytest tests/test_integration.py::test_survives_a_real_sigkill_exactly_once -v
```

What the test does:
1. Starts a subprocess running a workflow with a side effect (writes to a file).
2. Sends `SIGKILL` immediately after the side effect step commits.
3. Starts a new subprocess with the same arguments.
4. Asserts the side effect file shows the action happened **exactly once**.

Total: 1. That's the guarantee.
