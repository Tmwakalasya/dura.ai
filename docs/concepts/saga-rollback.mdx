## The other failure mode

Crash-and-resume handles the case where you want to *finish* the workflow.
But some failures aren't crashes — the flight has no seats, the API returns an
error, the inventory is out of stock. You don't want to retry; you want to
**undo what already happened** and surface a clean error.

Without saga rollback:

```
charge_card()   ← $412 taken from the customer
book_flight()   ← FAILS: no seats available
# charge is never refunded. customer is out $412 for a trip that isn't booked.
```

With dura, you register a compensating action alongside each step. If anything
goes wrong downstream, dura unwinds in reverse:

```
charge_card()   ← $412 charged
book_flight()   ← FAILS: no seats
  → undo: refund_card()   ← $412 returned. customer is made whole.
```

---

## The `undo=` argument

```python
@durable
def book_trip(ctx, dest, amount):
    charge = ctx.step(
        "charge_card",
        lambda: charge_card(amount),
        undo=lambda: refund_card(charge["id"]),   # compensating action
    )

    # If this raises for any reason, charge_card's undo fires automatically.
    seat = ctx.step(
        "book_flight",
        lambda: book_flight(dest),
        undo=lambda: cancel_hold(seat["hold_id"]),
    )

    # Safe to skip undo= for idempotent or non-reversible steps.
    ctx.step("send_confirmation", lambda: email.send(charge, seat))
```

The `undo` lambda captures the step's result from the enclosing scope, so you
always have access to the IDs and handles you need to reverse the operation.

---

## Unwind rules

**Only completed steps roll back.** The step that failed never had its side
effect succeed, so its `undo` does not fire. Dura only compensates steps that
actually ran.

**Reverse commit order.** If steps `a`, `b`, `c` completed before `d` fails,
the unwind runs `undo_c` → `undo_b` → `undo_a`. This matters when later steps
depend on earlier ones — releasing a seat reservation before refunding the
payment, for example.

**No `undo`, no-op.** A step without a compensating action is skipped during
unwind. Use this for idempotent operations or steps where "undo" doesn't make
sense (sending a read-only notification, incrementing a counter).

**Resume-safe.** A step replayed from the log (because it completed in a prior
run) still registers its `undo` on the saga stack. So if a crash happens during
step 3 and the process restarts, runs step 3 again, and then step 4 fails —
dura correctly rolls back steps 1, 2, and 3, even though 1 and 2 were committed
in a previous process.

**Best-effort unwind.** If an `undo` itself raises, dura records `"undo_failed"`
in `ctx.events` and continues unwinding the remaining steps. Partial rollback is
better than no rollback.

## Failure scenario table

For a three-step workflow — `charge_card`, `reserve_seat`, `confirm_booking`:

| Fails at | What completed before it | Unwind order |
|----------|--------------------------|--------------|
| `charge_card` | nothing | (no undo needed) |
| `reserve_seat` | `charge_card` | refund charge |
| `confirm_booking` | `charge_card`, `reserve_seat` | release seat → refund charge |

## Inspecting rollback in tests

After a call, `ctx.events` contains an ordered trail:

```python
book_trip("Tokyo", 412)  # book_flight raises

assert book_trip.last_context.events == [
    ("charge_card",  "executed"),
    ("book_flight",  "executed"),   # raised
    ("book_flight",  "rolled_back"),  # no undo= was set, so this is skipped
    ("charge_card",  "rolled_back"),
]

assert "charge_card" in book_trip.last_context.rolled_back
```

`rolled_back` is the list of step names whose `undo` successfully fired.
